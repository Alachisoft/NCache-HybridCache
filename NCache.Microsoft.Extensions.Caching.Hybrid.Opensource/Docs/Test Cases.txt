1. Initialization Scenarios
----------------------------
	1.1 Successful Initialization

		- INIT-002: Verify cache initializes correctly when configuration is bound from IConfiguration section → Expected: Configuration properties bound correctly, cache operational
		- INIT-003: Verify cache initializes correctly when configured using Action<NCacheHybridCacheConfiguration> delegate → Expected: Delegate invoked, options applied, cache initialized

		- INIT-005: Verify Pub/Sub topic is created and subscribed during initialization → Expected: Topic exists in NCache, subscription active
		- INIT-006: Verify TagKeeper is initialized with distributed cache reference → Expected: TagKeeper._distributedCache is not null
		- INIT-007: Verify unique identifier is generated using client IP and client ID → Expected: Identifier format matches "{IP}:{ClientId}"
		- INIT-008: Verify logging is enabled when EnableLogs is true and ILogger provided → Expected: Log entries written to configured sink
		- INIT-009: Verify NullLogger is used when EnableLogs is false → Expected: No log entries written
	1.2 Failed Initialization
		- INIT-010: Verify InvalidOperationException is thrown when LocalCacheName is null → Expected: InvalidOperationException with message "LocalCacheName is required."

		- INIT-012: Verify InvalidOperationException is thrown when DistributedCacheName is null → Expected: InvalidOperationException with message "DistributedCacheName is required."

		- INIT-015: Verify InvalidOperationException is thrown when ServerList is empty → Expected: InvalidOperationException with message "At least one server configuration is required."
		- INIT-016: Verify InvalidOperationException is thrown when ServerList contains null IP → Expected: InvalidOperationException with IP validation error
		- INIT-017: Verify InvalidOperationException is thrown when ServerList contains empty IP → Expected: InvalidOperationException with IP validation error

		- INIT-019: Verify InvalidOperationException is thrown when ServerList contains port < 1 → Expected: InvalidOperationException with port validation error
		- INIT-020: Verify InvalidOperationException is thrown when ServerList contains port > 65535 → Expected: InvalidOperationException with port validation error

2. GetOrCreateAsync Scenarios
------------------------------
	2.1 L1 Cache Hit Scenarios
		- GET-001: Verify value is returned from L1 cache when item exists and has no tags → Expected: item.GetValue<T>() returned, factory not invoked, latency < 1ms
		- GET-002: Verify value is returned from L1 cache when item exists and all tags are valid → Expected: item.GetValue<T>() returned, TagKeeper.IsItemInvalidatedAsync returns false
		- GET-003: Verify factory is NOT invoked when L1 cache hit occurs → Expected: Factory delegate invocation count = 0
		- GET-004: Verify L2 cache is NOT accessed when L1 cache hit occurs → Expected: No L2 Get operation logged, isItemInvalid flag not checked
		- GET-005: Verify correct type is returned when retrieving cached object from L1 → Expected: Returned object type matches expected type T via GetValue<T>()
	2.2 L2 Cache Hit Scenarios
		- GET-006: Verify value is returned from L2 cache when L1 misses but L2 hits → Expected: item.GetValue<T>() returned, factory not invoked
		- GET-007: Verify L1 cache is populated when value is fetched from L2 cache and tags are valid → Expected: Subsequent L1 Get returns the value
		- GET-008: Verify factory is NOT invoked when L2 cache hit occurs and tags are valid → Expected: Factory delegate invocation count = 0
		- GET-009: Verify L1 is populated with LocalCacheExpiration from options or config on L2 hit → Expected: L1 item has options.LocalCacheExpiration or config.LocalCacheExpiration applied
		- GET-010: Verify L1 population is skipped when DisableLocalCache flag is set on L2 hit → Expected: L1 Get returns null after L2 hit
		- GET-011: Verify L2 read is performed even when L1 item is tag-invalidated → Expected: L2 Get IS called (L2 may have fresher version), factory only invoked if L2 miss or L2 also invalid
	2.3 Cache Miss Scenarios
		- GET-012: Verify factory is invoked when item does not exist in L1 or L2 → Expected: Factory delegate invocation count = 1, cancellationToken passed
		- GET-013: Verify factory result is stored in L2 cache first after cache miss → Expected: L2 Get returns factory result with options.Expiration or config default
		- GET-014: Verify factory result is stored in L1 cache after L2 population → Expected: L1 Get returns factory result with options.LocalCacheExpiration or config default
		- GET-015: Verify factory state parameter is correctly passed to factory function → Expected: State value accessible inside factory(state, cancellationToken)
		- GET-016: Verify complex objects are correctly serialized and stored in cache → Expected: Deserialized object properties match original
		- GET-017: Verify null values returned by factory are cached correctly → Expected: Subsequent Get returns null without invoking factory
		- GET-018: Verify cache item is created with current Unix timestamp → Expected: NCacheHybridCacheItem.CreatedTimeStamp ≈ DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()
		- GET-019: Verify cache item stores associated tags → Expected: NCacheHybridCacheItem.Tags contains all provided tags
		- GET-020: Verify L2 and L1 population use separate expiration configurations → Expected: L2 has Expiration, L1 has LocalCacheExpiration
	2.4 Tag Invalidation During Get
		- GET-021: Verify L1 cached item is considered stale when its tag has been invalidated after item creation → Expected: isItemInvalid=true, factory invoked, fresh value returned
		- GET-022: Verify L2 cached item is considered stale when its tag has been invalidated after item creation → Expected: isItemInvalid=true (if L1 hit with invalid tags), factory invoked
		- GET-023: Verify item is invalidated when any one of its multiple tags is invalidated → Expected: IsItemInvalidatedAsync returns true
		- GET-024: Verify item is NOT invalidated when unrelated tag is invalidated → Expected: IsItemInvalidatedAsync returns false, cached value returned
		- GET-025: Verify all items are invalidated when wildcard (*) tag invalidation occurs → Expected: All GetOrCreateAsync calls invoke factory
		- GET-026: Verify item created AFTER tag invalidation is NOT considered stale → Expected: Cached value returned, factory not invoked
		- GET-027: Verify item created BEFORE global wildcard invalidation is considered stale → Expected: Factory invoked
		- GET-028: Verify item created AFTER global wildcard invalidation is NOT considered stale → Expected: Cached value returned
		- GET-029: Verify L2 is checked for fresher version even when L1 item is tag-invalidated → Expected: L2 accessed to check for newer timestamp before falling back to factory
	2.5 Cache Flags Scenarios
		- GET-030: Verify L1 cache read is skipped when DisableLocalCacheRead flag is set → Expected: L2 accessed even if L1 has value
		- GET-031: Verify L2 cache read is skipped when DisableDistributedCacheRead flag is set → Expected: Factory invoked even if L2 has value
		- GET-032: Verify L1 cache write is skipped when DisableLocalCacheWrite flag is set → Expected: L1 Get returns null after GetOrCreateAsync
		- GET-033: Verify L2 cache write is skipped when DisableDistributedCacheWrite flag is set → Expected: L2 Get returns null after GetOrCreateAsync
		- GET-034: Verify both L1 read and write are skipped when DisableLocalCache flag is set → Expected: L1 never accessed
		- GET-035: Verify both L2 read and write are skipped when DisableDistributedCache flag is set → Expected: L2 never accessed
		- GET-036: Verify all disable flags together causes factory invocation with no caching → Expected: Factory invoked, both caches empty
		- GET-037: Verify DisableUnderlyingData flag returns default(T) when key is null → Expected: default(T) returned, factory not invoked
		- GET-038: Verify null key without DisableUnderlyingData flag invokes factory → Expected: Factory invoked, result returned, no caching performed
	2.6 Expiration Scenarios
		- GET-039: Verify L2 cache item has Expiration applied from HybridCacheEntryOptions.Expiration → Expected: L2 CacheItem.Expiration.Type = Absolute, Duration = options.Expiration
		- GET-040: Verify L2 cache item has default Expiration from config when options.Expiration is null → Expected: L2 CacheItem.Expiration = config.DefaultEntryOptions.Expiration
		- GET-041: Verify L1 cache item has LocalCacheExpiration applied from HybridCacheEntryOptions.LocalCacheExpiration → Expected: L1 CacheItem.Expiration.Type = Absolute, Duration = options.LocalCacheExpiration
		- GET-042: Verify L1 cache item has default LocalCacheExpiration from config when options.LocalCacheExpiration is null → Expected: L1 CacheItem.Expiration = config.LocalCacheExpiration
		- GET-043: Verify L1 expires before L2 when LocalCacheExpiration < Expiration → Expected: L1 miss, L2 hit after LocalCacheExpiration elapsed
		- GET-044: Verify item is re-fetched from L2 after L1 expiration → Expected: L1 repopulated from L2 with LocalCacheExpiration
		- GET-045: Verify factory is invoked after both L1 and L2 items expire → Expected: Factory invoked, fresh value cached in L2 then L1
		- GET-046: Verify expiration precedence: options.Expiration overrides config default for L2 → Expected: L2 uses options value, not config
		- GET-047: Verify expiration precedence: options.LocalCacheExpiration overrides config default for L1 → Expected: L1 uses options value, not config
	2.7 Cache Stampede Prevention
		- GET-048: Verify semaphore lock prevents multiple concurrent factory invocations for same key → Expected: Factory invocation count = 1 for N concurrent requests
		- GET-049: Verify subsequent requests wait for first factory to complete and use cached result → Expected: All N requests return same value via item.GetValue<T>()
		- GET-050: Verify semaphore WaitAsync accepts cancellationToken parameter → Expected: await lock.WaitAsync(cancellationToken) called
		- GET-051: Verify semaphore is released in TRY-FINALLY block after factory completes successfully → Expected: _reqLocks.TryRemove(key, out _) in finally, no deadlock
		- GET-052: Verify semaphore is released in TRY-FINALLY block after factory throws exception → Expected: _reqLocks.TryRemove(key, out _) in finally, exception propagated
		- GET-053: Verify different keys can be processed concurrently without blocking each other → Expected: Concurrent GetOrCreateAsync for different keys complete independently
		- GET-054: Verify L1 is re-checked after acquiring semaphore lock (double-check pattern) → Expected: Second request uses item.GetValue<T>() from L1 populated by first request
	2.8 Error Scenarios
		- GET-055: Verify null key with DisableUnderlyingData flag returns default(T) → Expected: default(T) returned, no exception
		- GET-056: Verify null key without DisableUnderlyingData flag invokes factory and returns result → Expected: Factory result returned, no caching, no exception
		- GET-057: Verify empty key is treated same as null key → Expected: Same behavior as null key (factory or default(T))
		- GET-058: Verify factory exception propagates to caller unchanged → Expected: Same exception type and message thrown to caller
		- GET-059: Verify InvalidCastException is thrown when retrieving item with mismatched type → Expected: InvalidCastException from GetValue<T>() with type mismatch
		- GET-060: Verify OperationCanceledException is thrown when cancellation token is triggered before factory completes → Expected: OperationCanceledException from factory(state, cancellationToken)
		- GET-061: Verify OperationCanceledException is thrown when cancellation token is triggered during semaphore wait → Expected: OperationCanceledException from lock.WaitAsync(cancellationToken)
		- GET-062: Verify partial cache state is not corrupted when factory throws exception → Expected: Caches remain in pre-call state, semaphore released in finally
		- GET-063: Verify semaphore cleanup occurs even when exception thrown → Expected: _reqLocks.TryRemove called in finally block
